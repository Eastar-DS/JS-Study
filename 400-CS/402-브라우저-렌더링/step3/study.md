# step3 — CSS가 CSSOM이 되기까지

### 정리
CSSOM 없이 CSS를 그냥 "스타일 규칙 목록"으로만 알면 흔히 이런 오해가 생깁니다.

"CSS는 그냥 위에서 아래로 마지막 값이 이기는 거 아닌가?"
"DOM에 스타일 정보가 같이 들어있는 거 아닌가?"
"CSS 파싱은 단순히 문자열 자르기 아닌가?"
step3는 이 셋을 깨고 — 우선순위 / 상속 / 기본 스타일 때문에 CSS도 별도의 객체 모델로 정리되어야 한다는 감각을 박는 단계입니다.


### 질문
- CSS 규칙에서 선택자, 속성, 값은 각각 무엇인가요?
=> 선택자는 속성과 값을 적용할 대상, 속성은 width같은 속성, 값은 속성에 할당할 값.

- 하나의 요소에 여러 CSS 규칙이 적용될 수 있다는 말은 무슨 뜻인가요?
=> 태그, 클래스이름, id로 같은 대상을 지정할 수 있는 등 하나의 요소에 여러 규칙이 적용될 수 있다.

- CSS 우선순위와 상속 때문에 브라우저가 추가 계산을 해야 하는 이유는 무엇인가요?
=> 부모태그의 css 규칙을 자식태그에도 적용하는 등 여러 규칙이 적용될 수 있어 html처럼 단순히 위에서 아래로 해석하면 안됨.

-  CSSOM은 무엇인가요?
=> HTML에 적용할 CSS 문자열을 객체로 변환한 것.

-  DOM과 CSSOM은 각각 어떤 정보를 담당하나요?
=> DOM : 구조, CSSOM : 꾸미는 정보

-  CSS가 렌더링에 중요한 이유는 무엇인가요?
=> 어떤 태그에 display:none 속성이 있었다고 한다면 DOM만가지고 먼저 랜더링을 한다고 가정했을때 화면이 깜빡이는등의 현상이 발생함. 랜더링전에 반드시 알고있어야 하는 정보임.


Q1. CSS도 "파싱"이 필요한 이유를 한 문장으로 적어보세요.
    (힌트: CSS도 처음에는 문자열로 도착합니다. 브라우저가 화면을 그리기 전에
     무엇을 알아야 하는지 떠올려 보세요.)
=> 문자열에서 변경된 정보를 가져오는건 어려움. DOM을 만들었을 때 처럼 객체로 변환해두는게 접근성이 좋음.

Q2. 아래 HTML/CSS에서 button의 최종 color는 무엇이고, 브라우저는 그 결정을 위해
    무엇을 계산해야 하나요?

    <button id="save" class="primary">저장</button>

    button { color: black; }
    .primary { color: white; }
    #save { color: yellow; }

    (정답을 외우는 게 목적이 아니라 "왜 단순한 마지막 값이 아닌가"를 답하는 게 핵심)
=> 노란색. 버튼태그의 색상, 클래스와 id의 색상 속성을 확인하고 가장 좁은 범위인 id의 색상 속성을 적용함.

Q3. DOM과 CSSOM을 따로 만드는 게 왜 합리적인가요?
    하나의 객체 트리로 합쳐 만드는 것보다 무엇이 좋을까 추측해 보세요.
    (이건 추측 문제 — 정답이 아니라 본인 추론이 목적)
=> html문자열과 css문자열은 별개의 역할로 구분되어있으므로 파싱하는 과정을 동시에 진행할 수 있어서?

점검 질문 (스스로에게)
step2에서 본인이 던진 질문 "값을 바꿀 뿐인데 레이아웃이 뭐길래 오랜 시간?" — 이번 step3 학습 후, 그 질문에 어떤 부분 답을 추가할 수 있을지 한 줄 적어두세요. (아직 완전한 답은 아닐 거예요. 그게 정상)
=> 레이아웃이 변경되면 html에비해 파싱과정이 오래걸리는 CSSOM객체트리를 계속해서 다시 생성해야하기때문에? (근데 그러면 페인트나 컴포짓과정도 비슷하게 시간이 걸려야하는거 아닌가? 왜 레이아웃이 자원이 많이 소모되는거지)
