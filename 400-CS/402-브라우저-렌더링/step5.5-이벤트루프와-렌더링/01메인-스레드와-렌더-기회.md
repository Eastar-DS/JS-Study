# 메인 스레드와 렌더 기회

## 한 문장으로

브라우저는 JavaScript 실행과 렌더링을 같은 메인 스레드에서 번갈아 처리하기 때문에, JS가 길게 실행되는 동안에는 화면을 다시 그릴 기회가 없습니다.

## 왜 이걸 알아야 할까?

`step5`에서 JavaScript가 DOM과 스타일을 바꾸면 렌더링이 다시 필요할 수 있다는 것을 봤습니다.

그런데 한 가지 질문이 남습니다.

```text
브라우저는 도대체 언제 화면을 다시 그릴까?
```

이 질문에 답하려면 이벤트 루프와 렌더링이 같은 스레드 위에서 어떻게 자리를 나눠 쓰는지 알아야 합니다.

## 메인 스레드는 한 번에 하나만 한다

브라우저의 메인 스레드는 같은 시점에 한 가지 일만 합니다.

```text
JavaScript를 실행하거나
스타일을 계산하거나
Layout을 계산하거나
Paint 명령을 만들거나
```

이 일들이 동시에 진행되지 않고 순서대로 처리됩니다.

그래서 JavaScript가 길게 실행되면 그 시간 동안 다른 일은 멈춰 있습니다.

```js
// 이 함수가 끝날 때까지 화면은 다시 그려지지 않는다
function blockMainThread() {
  const start = Date.now();
  while (Date.now() - start < 3000) {
    // 3초 동안 메인 스레드를 잡고 있음
  }
}

button.addEventListener("click", () => {
  blockMainThread();
});
```

버튼을 누르면 화면이 3초 동안 멈춥니다.

스크롤도, 애니메이션도, 마우스 호버 효과도 반응하지 않습니다.

## 한 프레임 안에 일어나는 일

브라우저는 보통 초당 60번 정도 화면을 갱신합니다.

한 프레임에 쓸 수 있는 시간은 대략 16.7ms입니다.

이 짧은 시간 안에 메인 스레드는 대략 이런 일을 합니다.

```text
1. Task 하나 실행 (JS 코드 또는 이벤트 핸들러)
2. Microtask Queue 비우기 (Promise.then 같은 작업)
3. 필요하면 화면 다시 그리기
   - 스타일 계산
   - Layout
   - Paint
   - Composite
4. 다음 Task로 이동
```

이 흐름에서 "필요하면 화면 다시 그리기"가 일어나는 시점을 **렌더 기회(rendering opportunity)** 라고 부릅니다.

## 렌더 기회란

렌더 기회는 브라우저가 "이번 프레임에 화면을 다시 그릴 수 있을 것 같다"고 판단하는 시점입니다.

중요한 것은 두 가지입니다.

1. 렌더 기회는 매 Task가 끝날 때마다 무조건 오는 것이 아닙니다.
2. 렌더 기회가 오려면 Call Stack이 비어 있어야 합니다.

```text
JS Task 실행 -> Stack에 함수가 쌓임
함수 끝남 -> Stack 비워짐
Microtask Queue 비움
브라우저: "이제 한 프레임 시간이 남았나? 그릴 게 있나?"
필요하면 렌더링 단계 진행
```

JS Task가 16.7ms를 넘게 잡고 있으면 이번 프레임에는 렌더 기회가 없을 수 있습니다.

그래서 화면이 멈춰 보이는 것입니다.

## 마이크로태스크는 렌더링 전에 실행된다

이스타가 이미 공부한 마이크로태스크 큐는 렌더 기회와 어떻게 다를까요?

```text
1. 현재 Task 실행 종료
2. Microtask Queue가 비워질 때까지 마이크로태스크 실행
3. (필요하면) 렌더링 단계 진행
4. 다음 Task로 이동
```

즉, 마이크로태스크는 **렌더링 직전**에 모두 실행됩니다.

그래서 마이크로태스크가 무한히 쌓이면 렌더 기회가 영영 오지 않습니다.

```js
function blockWithMicrotasks() {
  Promise.resolve().then(blockWithMicrotasks);
}

blockWithMicrotasks();
```

이 코드는 Stack은 비우지만, Microtask Queue가 끝없이 차오릅니다. 결과적으로 화면은 멈춥니다.

## 흔한 오해

### 오해 1. setTimeout(fn, 0)을 쓰면 즉시 화면이 다시 그려진다

아닙니다.

`setTimeout(fn, 0)`은 fn을 다음 Task로 미루는 효과가 있습니다.

그래서 현재 Task가 끝난 뒤 마이크로태스크와 렌더 기회를 거친 다음에야 fn이 실행될 수 있습니다.

이 사이에 렌더 기회가 끼어들 수 있다는 것이지, fn이 즉시 실행되는 것은 아닙니다.

### 오해 2. JS가 빠르면 무조건 60fps가 나온다

아닙니다.

JS 실행 시간뿐 아니라 스타일 계산, Layout, Paint, Composite도 모두 16.7ms 안에 끝나야 합니다.

JS가 5ms로 끝나도 Layout이 무거운 변경을 만들면 한 프레임을 넘길 수 있습니다.

## 가상 DOM, Fiber로의 복선

이 챕터에서 기억해야 할 한 문장은 이것입니다.

> "JS Task 하나가 길어지면 그 동안 화면은 멈춘다."

가상 DOM과 React Fiber는 이 문제를 정면으로 다룹니다.

- 가상 DOM: 렌더링 비용을 줄이기 위해 변경을 메모리에서 미리 비교한다.
- Fiber: 그 비교 작업조차 한 번에 끝내지 않고 작은 단위로 쪼개서, 중간중간 렌더 기회를 브라우저에 양보한다.

지금은 "왜 작업을 쪼개야 하는가"의 답이 메인 스레드 점유 문제에 있다는 것만 기억하면 됩니다.
