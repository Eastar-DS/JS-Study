# 렌더링 성능 기본

## 한 문장으로

렌더링 성능 최적화는 브라우저가 다시 계산해야 하는 렌더링 단계를 줄이고, 한 프레임 안에 일을 끝낼 수 있게 만드는 것입니다.

## 왜 성능을 공부해야 할까?

사용자는 화면이 부드럽게 반응하기를 기대합니다.

버튼을 눌렀는데 화면이 늦게 바뀌거나, 스크롤이 버벅이거나, 애니메이션이 끊기면 사용자는 페이지가 느리다고 느낍니다.

렌더링 성능은 이런 경험과 직접 연결됩니다.

## 프레임 감각

많은 화면은 초당 60번 정도 갱신되는 것을 목표로 합니다.

초당 60프레임이면 한 프레임에 쓸 수 있는 시간은 대략 16.7ms입니다.

```text
1000ms / 60fps = 약 16.7ms
```

이 시간 안에 JavaScript 실행, 스타일 계산, Layout, Paint, Composite 같은 일이 끝나야 부드럽게 보입니다.

항상 16.7ms를 외울 필요는 없지만, 브라우저가 한 화면 갱신에 쓸 수 있는 시간이 매우 짧다는 감각은 중요합니다.

## 성능을 나쁘게 만들 수 있는 코드

### 1. 반복문 안에서 DOM을 계속 추가하기

```js
for (let i = 0; i < 1000; i++) {
  const li = document.createElement("li");
  li.textContent = `item ${i}`;
  list.appendChild(li);
}
```

브라우저가 최적화해주는 경우도 있지만, 많은 DOM 변경은 렌더링 비용을 키울 수 있습니다.

개선 방향은 여러 변경을 모아서 한 번에 반영하는 것입니다.

### 2. 스타일 쓰기와 레이아웃 읽기를 반복하기

```js
for (const box of boxes) {
  box.style.width = "300px";
  console.log(box.offsetWidth);
}
```

스타일을 바꾸고 바로 레이아웃 값을 읽으면 브라우저가 최신 값을 계산하기 위해 Layout을 강제로 수행할 수 있습니다.

개선 방향은 읽기와 쓰기를 분리하는 것입니다.

```js
const widths = Array.from(boxes).map((box) => box.offsetWidth);

for (const box of boxes) {
  box.style.width = "300px";
}
```

### 3. Layout을 유발하는 속성으로 애니메이션하기

```css
.box {
  left: 100px;
}
```

`left` 변경은 위치 계산을 다시 필요로 할 수 있습니다.

가능하면 움직임은 `transform`으로 표현하는 것이 유리한 경우가 많습니다.

```css
.box {
  transform: translateX(100px);
}
```

## 최적화의 기본 원칙

```text
1. DOM 변경을 줄인다.
2. 변경을 여러 번 나누지 말고 묶는다.
3. Layout을 다시 계산하게 만드는 속성을 조심한다.
4. 애니메이션은 가능하면 transform, opacity를 우선 고려한다.
5. 느낌으로 판단하지 말고 DevTools로 관찰한다.
```
