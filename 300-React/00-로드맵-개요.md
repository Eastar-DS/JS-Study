# React 학습 로드맵 개요

이 문서는 React를 처음 실전 프로젝트로 접한 뒤, 개념을 체계적으로 다시 쌓기 위한 학습 로드맵입니다.

## 이 로드맵의 기준

- React 내부 구현보다 먼저 React의 사용자 모델을 이해합니다.
- "왜 리렌더링되는가", "이 상태는 어디에 둬야 하는가", "effect가 정말 필요한가"를 우선 해결합니다.
- 각 단계는 `목표 -> 읽을 것 -> 미션 -> 결제 프로젝트 연결 연습` 순서로 진행합니다.
- 현재 프로젝트 `react-payments`를 반복 실험장으로 사용합니다.

## 포함한 핵심 주제

아래 주제들은 기존에 이야기한 항목과, 실제로 빠지면 곧바로 막히는 주제를 합쳐 정리한 것입니다.

1. React mental model: render, commit, tree
2. state snapshot, batching, functional update
3. key, identity, state 보존과 초기화
4. state / ref / URL / derived state 구분
5. effect가 필요한 경우와 필요 없는 경우
6. JavaScript 관점의 React: closure, reference, stale value
7. 관심사 분리: component / hook / utility
8. 재사용 가능한 컴포넌트와 form 설계
9. state 공유: lifting state up, props drilling, context
10. 리렌더링 추적과 성능 최적화
11. virtual DOM, diffing, reconciliation, key
12. hooks 내부 원리, Fiber, Suspense, transition
13. 상태관리 도구 흐름: Context, Redux, Zustand, Jotai

## 기존 목록에 없었지만 반드시 넣은 주제

아래는 이전 대화에서 직접 이름이 크지 않았지만, 학습 계획에 꼭 들어가야 한다고 판단한 주제입니다.

- `State as a Snapshot`
- batching과 함수형 업데이트
- `You Might Not Need an Effect`
- derived state와 duplicated state
- controlled / uncontrolled input
- component identity와 state preservation/reset
- render purity와 immutability

이 주제들은 실제로 다음 문제들을 풀기 위해 필요합니다.

- 왜 `setState` 직후 값이 안 바뀐 것처럼 보이는가
- 왜 같은 컴포넌트인데 state가 유지되기도 하고 초기화되기도 하는가
- 왜 굳이 state로 저장하지 말고 계산해야 하는 값이 있는가
- 왜 effect가 남용되면 코드가 더 꼬이는가
- 왜 form 컴포넌트 설계가 계속 어려운가

## 권장 진행 방식

- 각 step의 문서를 읽고 바로 코드 실험을 합니다.
- 문서를 한 번에 다 읽지 말고, 한 step씩 완료합니다.
- 각 step의 미션은 가능한 한 현재 `react-payments` 코드에서 실험합니다.
- 학습 로그가 필요하면 별도 문서로 남깁니다.
