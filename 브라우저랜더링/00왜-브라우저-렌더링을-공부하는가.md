# 왜 브라우저 렌더링을 공부하는가

## 한 문장으로

브라우저 렌더링을 이해하면 HTML, CSS, JavaScript가 실제 화면으로 바뀌는 비용을 이해하게 됩니다.

그래서 화면이 느리거나 버벅일 때 원인을 찾을 수 있고, 어떤 코드를 바꾸면 Layout, Paint, Composite 중 무엇이 다시 일어나는지 판단할 수 있습니다.

결과적으로 성능 문제를 감으로 고치는 것이 아니라 근거를 갖고 고칠 수 있습니다.

## 코드에서 무엇을 이해할 수 있을까?

예를 들어 이런 코드를 봅시다.

```js
element.style.width = "300px";
```

브라우저 렌더링을 모르면 이 코드는 "스타일 하나 바꾸는 코드"로만 보입니다.

브라우저 렌더링을 알면 이렇게 보입니다.

```text
width가 바뀐다.
-> 요소의 크기가 바뀐다.
-> 주변 요소 위치도 바뀔 수 있다.
-> Layout/Reflow가 다시 필요할 수 있다.
-> 이후 Paint, Composite도 이어질 수 있다.
```

그래서 반복문 안에서 이런 코드가 많으면 성능에 위험할 수 있다고 판단할 수 있습니다.

```js
items.forEach((item) => {
  item.style.width = `${item.offsetWidth + 10}px`;
});
```

이 코드는 특히 조심해야 합니다.

`offsetWidth`는 레이아웃 계산 결과를 읽는 값입니다.

`style.width`는 레이아웃을 바꾸는 쓰기 작업입니다.

읽기와 쓰기가 반복해서 섞이면 브라우저가 레이아웃 계산을 여러 번 강제로 수행할 수 있습니다.

## CSS에서 무엇을 이해할 수 있을까?

브라우저 렌더링을 이해하면 CSS 속성을 단순히 "예쁘게 만드는 속성"이 아니라 "브라우저의 어떤 계산을 유발하는 속성"으로 볼 수 있습니다.

### Layout과 관련이 큰 속성

```css
.card {
  width: 300px;
  height: 200px;
  margin: 20px;
}
```

`width`, `height`, `margin`은 요소의 크기나 위치에 영향을 줍니다.

그래서 Layout 계산과 관련이 큽니다.

### Paint와 관련이 큰 속성

```css
.card {
  background-color: red;
  box-shadow: 0 4px 20px rgba(0, 0, 0, 0.2);
}
```

`background-color`, `box-shadow`는 요소가 어떻게 칠해지는지와 관련이 큽니다.

크기와 위치가 그대로여도 색이나 그림자를 다시 그려야 할 수 있습니다.

그래서 Paint와 관련이 큽니다.

### Composite와 관련이 큰 속성

```css
.card {
  transform: scale(1.05);
  opacity: 0.8;
}
```

`transform`, `opacity`는 Layout이나 Paint를 다시 하지 않고 Composite 단계에서 처리될 수 있는 경우가 많습니다.

그래서 애니메이션을 만들 때 자주 권장됩니다.

```css
/* Layout에 영향을 줄 가능성이 큼 */
.box {
  left: 100px;
}

/* Composite 단계에서 처리될 가능성이 큼 */
.box {
  transform: translateX(100px);
}
```

이 차이를 알면 왜 어떤 hover 효과는 부드럽고, 어떤 hover 효과는 버벅이는지 설명할 수 있습니다.

## 실무 버그를 어떻게 다르게 볼 수 있을까?

실무에서는 이런 문제가 자주 나옵니다.

```text
페이지가 처음 뜰 때 레이아웃이 밀린다.
이미지가 늦게 로드되면서 아래 콘텐츠가 튄다.
모달을 열 때 화면이 버벅인다.
스크롤이 끊긴다.
애니메이션이 부드럽지 않다.
버튼 클릭 후 화면 반응이 늦다.
```

브라우저 렌더링을 모르면 전부 "느리다", "버벅인다", "튄다" 정도로만 보입니다.

브라우저 렌더링을 알면 문제를 더 구체적으로 나눌 수 있습니다.

```text
레이아웃이 밀림
-> Layout 계산 결과가 나중에 바뀌었나?

이미지 때문에 화면이 튐
-> 이미지 크기 정보가 미리 없었나?

애니메이션이 버벅임
-> Layout이나 Paint를 계속 유발하나?

클릭 반응이 늦음
-> JavaScript가 메인 스레드를 오래 잡고 있나?

화면이 깜빡임
-> CSSOM, 폰트, 이미지 로딩 타이밍 문제인가?
```

즉, "느리다"를 "어느 단계가 비싼가"로 바꿔서 볼 수 있습니다.

## DevTools에서 무엇을 이해할 수 있을까?

브라우저 렌더링을 모르면 Performance 탭을 봐도 막대가 많고 복잡해 보입니다.

브라우저 렌더링을 알면 DevTools의 항목들이 의미를 갖습니다.

```text
Scripting: JavaScript 실행 시간
Rendering: 스타일 계산, Layout
Painting: 픽셀을 칠하는 작업
Composite Layers: 레이어 합성
```

예를 들어 Performance 탭에서 `Layout`이 반복적으로 많이 보이면 아래 가능성을 생각할 수 있습니다.

```text
DOM 크기나 위치 변경이 많을 수 있다.
레이아웃 값을 읽는 코드가 반복될 수 있다.
CSS 변경이 주변 요소를 계속 밀어내고 있을 수 있다.
```

`Paint`가 크면 아래 가능성을 생각할 수 있습니다.

```text
box-shadow, filter, 큰 이미지, 복잡한 배경이 문제일 수 있다.
넓은 영역의 색상 변경이 자주 일어날 수 있다.
```

`Scripting`이 크면 아래 가능성을 생각할 수 있습니다.

```text
렌더링 이전에 JavaScript가 너무 오래 실행되고 있을 수 있다.
브라우저가 화면을 그려야 하는데 JavaScript 때문에 기다리고 있을 수 있다.
```

이렇게 DevTools를 단순히 그래프를 보는 도구가 아니라 원인을 추적하는 도구로 사용할 수 있습니다.

## 실무에서 가장 직접적인 이득

1. 성능 문제를 감이 아니라 근거로 분석할 수 있습니다.
2. CSS 속성 선택을 성능 관점에서 할 수 있습니다.
3. 애니메이션을 더 부드럽게 만들 수 있습니다.
4. Layout Shift 같은 화면 튐 문제를 줄일 수 있습니다.
5. DevTools Performance 패널을 읽을 수 있습니다.
6. React, Vue 같은 프레임워크의 렌더링 결과가 실제 브라우저 렌더링과 어떻게 연결되는지 이해할 수 있습니다.
7. 코드 리뷰에서 "이 코드는 왜 위험한지"를 설명할 수 있습니다.

## 다른 사람에게 설명할 때

아직 브라우저 렌더링을 모르는 사람에게는 이렇게 말할 수 있습니다.

```text
브라우저 렌더링을 이해하면 화면이 느린 이유를 브라우저의 작업 단계로 쪼개서 볼 수 있어요.

어떤 코드가 Layout을 다시 만들고, 어떤 코드가 Paint를 다시 만들고, 어떤 코드는 Composite만으로 끝날 수 있는지 판단할 수 있습니다.

그래서 성능 개선, 애니메이션 최적화, 레이아웃 문제 해결, DevTools 분석 능력이 전부 좋아집니다.
```
