## 쉬운 설명부터

학교를 상상해보세요.

- **Code 영역** = **교과서 보관함**: 수업 내용(명령어)이 적혀있는 곳. 학생이 바꿀 수 없고, 읽기만 합니다.
- **Data 영역** = **게시판**: 학기 내내 바뀌지 않는 공지사항(전역 변수, 상수)이 붙어있는 곳. 프로그램 시작부터 끝까지 유지됩니다.
- **Stack 영역** = **칠판**: 수업 중에 풀이 과정을 적었다가 다 풀면 지우는 곳. 함수가 호출되면 적고, 끝나면 지웁니다.
- **Heap 영역** = **사물함**: 크기가 다양한 물건(객체)을 자유롭게 넣고 빼는 곳. 언제 넣고 뺄지 미리 정해져 있지 않습니다.

## 자세한 설명

프로그램이 실행되면, 운영체제는 해당 프로세스에게 메모리를 할당합니다. 이 메모리는 크게 4가지 영역으로 나뉩니다.

```
┌──────────────────────────────────────┐  높은 주소
│              Stack 영역               │  ↓ 아래로 자란다
│   (함수 호출, 지역 변수, 매개변수)       │
│                  ↓                    │
│                                      │
│                  ↑                    │
│              Heap 영역                │  ↑ 위로 자란다
│   (동적 할당, 객체, 배열)               │
├──────────────────────────────────────┤
│              Data 영역                │
│   (전역 변수, 정적 변수)               │
├──────────────────────────────────────┤
│              Code 영역                │
│   (실행할 기계어 명령어)               │
└──────────────────────────────────────┘  낮은 주소
```

### 1. Code 영역 (Text Segment)

**"무엇을 실행할 것인가"가 저장되는 곳**

```js
function add(a, b) {
  return a + b;
}
```

이 함수의 코드 자체(CPU가 읽을 기계어로 컴파일된 명령어)가 Code 영역에 저장됩니다.

특징:

- **읽기 전용(Read-Only)** — 실행 중에 코드가 변경되면 안 되므로
- 프로그램 시작 시 한 번 로드되고, 종료될 때까지 유지
- 같은 프로그램을 여러 번 실행해도 Code 영역은 공유 가능 (메모리 절약)

### 2. Data 영역 (Data Segment)

**"프로그램 전체에서 살아있는 값"이 저장되는 곳**

```js
// 전역 변수 — 프로그램 시작부터 끝까지 존재
const MAX_RETRY = 3;
let currentUser = null;
```

특징:

- **프로그램 시작 시 할당, 종료 시 해제**
- 전역 변수, 정적(static) 변수가 여기에 위치
- 더 세분화하면:
    - **.data** — 초기값이 있는 전역 변수 (`let count = 0`)
    - **.bss** — 초기값이 없는 전역 변수 (`let result`) → 0으로 초기화

### 3. Stack 영역 (Call Stack)

**"지금 실행 중인 함수의 맥락"이 저장되는 곳**

이벤트 루프를 공부하면서 만난 **Call Stack**이 바로 이 메모리 영역입니다.

```js
function multiply(x, y) {
  const result = x * y;  // result → Stack에 저장
  return result;
}

function calculate() {
  const a = 10;          // a → Stack에 저장
  const b = 20;          // b → Stack에 저장
  return multiply(a, b);
}

calculate();
```

함수가 호출될 때마다 **스택 프레임(Stack Frame)**이 쌓입니다:

```
호출 시:                          반환 시:

│                    │           │                    │
│  multiply 프레임    │  ← top    │                    │
│  x=10, y=20       │           │                    │
│  result=200       │           │                    │
├────────────────────┤           │                    │
│  calculate 프레임   │           │  calculate 프레임   │  ← top
│  a=10, b=20       │           │  a=10, b=20       │
├────────────────────┤           ├────────────────────┤
│  전역 프레임         │           │  전역 프레임         │
└────────────────────┘           └────────────────────┘
```

특징:

- **LIFO (Last In, First Out)** — 마지막에 들어간 것이 먼저 나옴
- 함수 호출 → 프레임 push / 함수 반환 → 프레임 pop
- **크기가 제한적** — 재귀가 너무 깊으면 `RangeError: Maximum call stack size exceeded`
- **자동 관리** — 함수가 끝나면 해당 프레임의 메모리가 자동으로 해제됨

### 4. Heap 영역

**"크기와 생존 기간을 미리 알 수 없는 값"이 저장되는 곳**

JavaScript에서 **객체, 배열, 함수 객체** 등 참조 타입의 실제 데이터가 여기 저장됩니다.

```js
function createUser() {
  // user 객체의 실제 데이터 → Heap에 저장
  // user 변수(Heap 주소를 담은 참조값) → Stack에 저장
  const user = {
    name: '김개발',
    skills: ['JS', 'React', 'Node']
  };
  return user;
}

const dev = createUser();
// createUser의 스택 프레임은 사라졌지만,
// Heap에 있는 { name: '김개발', ... } 객체는 살아있다!
// dev가 그 Heap 주소를 참조하고 있으므로.
```

특징:

- **동적 할당** — 런타임에 필요한 만큼 할당
- **비구조적** — Stack처럼 순서가 없고, 여기저기 흩어져 저장됨
- **가비지 컬렉션(GC)으로 관리** — 더 이상 참조되지 않는 객체를 자동 해제
- **크기 제한이 Stack보다 훨씬 큼** — 대용량 데이터 저장 가능

## Stack vs Heap: JavaScript 관점에서 핵심 구분

```js
// === 기본형 (Primitive) → 값 자체가 Stack에 ===
let age = 25;           // Stack: age = 25
let name = 'Lee';       // Stack: name = 'Lee'

// === 참조형 (Reference) → 참조(주소)는 Stack, 실제 데이터는 Heap ===
let scores = [90, 85];  // Stack: scores = 0x001 (주소)
                         // Heap[0x001]: [90, 85] (실제 배열)

let person = {           // Stack: person = 0x002 (주소)
  age: 25               // Heap[0x002]: { age: 25 } (실제 객체)
};
```

이것을 그림으로 보면:

```
        Stack                          Heap
  ┌──────────────┐            ┌─────────────────┐
  │ age    │  25  │            │                 │
  ├────────┼──────┤            │  0x001           │
  │ name   │'Lee' │            │  ┌─────────────┐│
  ├────────┼──────┤            │  │ [90, 85]    ││
  │ scores │0x001 │───────────→│  └─────────────┘│
  ├────────┼──────┤            │                 │
  │ person │0x002 │──────┐     │  0x002           │
  └────────┴──────┘      │     │  ┌─────────────┐│
                          └────→│  │ { age: 25 } ││
                               │  └─────────────┘│
                               └─────────────────┘
```

## 이벤트 루프 맥락에서 왜 이게 중요한가

키워드 학습 로드맵의 1단계에서 Heap이 등장하는 이유를 생각해보세요:

```js
console.log('시작');

setTimeout(() => {
  // 이 화살표 함수 객체는 어디에 저장될까?
  // 타이머 완료 후 이 콜백을 Task Queue에 넣으려면,
  // 이 함수 객체가 사라지면 안 된다.
  console.log('타이머 완료');
}, 1000);

console.log('끝');
// 이 시점에서 현재 실행 컨텍스트의 Stack 프레임은 정리되지만,
// setTimeout에 전달한 콜백 함수는 여전히 ???에 살아있다.
```

**생각해볼 것**: `setTimeout`의 콜백 함수가 1초 뒤에 실행되려면, 현재 코드 실행이 끝나도 그 함수 객체가 메모리 어딘가에 살아있어야 합니다. Stack은 함수가 끝나면 정리되는데... 그렇다면 이 콜백 함수 객체는 4개 영역 중 어디에 저장되어 있기에 나중에 실행할 수 있는 걸까요?

이 질문에 답할 수 있다면, Call Stack과 Heap의 역할 분담, 그리고 이벤트 루프가 이 메모리 구조 위에서 어떻게 동작하는지가 자연스럽게 연결될 것입니다.

공부 전 : 
코드 : 함수들을 기계어로 등록하는 곳
데이터 : 변수들 저장해 놓는 곳
힙 메모리 : 데이터에 주소가 저장되어있는 경우 실제 데이터들을 저장하는 곳
콜 스택 : 실행 순서를 등록하는 곳
문제 답 : 코드 영역에 살아있을 것 같다.

## 추천 학습 방법

현재 **Data와 Stack의 차이**가 가장 흐릿한 상태이고, **Heap은 감을 잡고 있습니다.** 이 상태에서 추천하는 순서:

**1단계 — 아래 코드에서 각 변수가 어디에 저장되는지 직접 분류해보세요:**

```js
const API_URL = 'https://api.com';   // 이건 어디?

function fetchUser() {
  const userId = 42;                  // 이건 어디?
  const user = { name: '김개발' };    // user 변수는? { name: '김개발' } 객체는?
  return user;
}

const result = fetchUser();           // fetchUser 끝난 후, { name: '김개발' }은 어디에?
```

**2단계 — 분류한 후 자료의 "Stack vs Heap: JavaScript 관점에서 핵심 구분" 섹션을 다시 읽으세요.** 특히 기본형(Primitive)은 Stack에, 참조형(Reference)은 Heap에라는 구분을 코드에 대입해보면서요.

**3단계 — Data 영역을 다시 읽을 때, 핵심 질문은 "생명 주기"입니다:**

```
Data: 프로그램 시작 ~ 종료까지 살아있는 변수 (전역)
Stack: 함수 호출 ~ 함수 반환까지 살아있는 변수 (지역)
Heap: 누군가 참조하는 동안 살아있는 객체 (동적)
```

"어디에 저장되는가"보다 **"언제 생기고 언제 사라지는가"**로 구분하면 훨씬 명확해질 것입니다.
