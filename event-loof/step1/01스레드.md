## 쉬운 설명부터

식당을 상상해보세요.

**스레드는 식당의 요리사**입니다. 요리사가 1명이면 한 번에 하나의 요리만 만들 수 있고, 요리사가 4명이면 동시에 4개의 요리를 만들 수 있습니다.

컴퓨터 프로그램에서 "일을 처리하는 일꾼" 하나하나가 바로 스레드입니다.

## 자세한 설명

### 프로세스와 스레드의 관계

먼저 **프로세스(Process)**를 알아야 스레드를 이해할 수 있습니다.

- **프로세스**: 운영체제로부터 메모리를 할당받아 실행 중인 프로그램의 단위. 크롬 브라우저를 열면 하나의 프로세스가 생깁니다.
- **스레드**: 프로세스 내부에서 실제로 코드를 실행하는 흐름의 단위. 하나의 프로세스 안에 여러 스레드가 존재할 수 있습니다.

```
[프로세스] ── 메모리 공간 (Code, Data, Heap)
   │
   ├── 스레드 1 (자체 Call Stack 보유)
   ├── 스레드 2 (자체 Call Stack 보유)
   └── 스레드 3 (자체 Call Stack 보유)
         ↑
   각 스레드는 독립적인 실행 흐름을 갖지만,
   같은 프로세스의 메모리(Heap)는 공유한다
```

[[012. 메모리 공간 (Code, Data, Heap, Stack)]]
핵심은 이것입니다:

- 각 스레드는 **자기만의 Call Stack**을 가집니다 (독립적 실행 흐름)
- 하지만 **Heap 메모리는 공유**합니다 (같은 데이터에 접근 가능)

### 싱글 스레드 vs 멀티 스레드

**멀티 스레드** 언어 (Java, C++ 등):

```
스레드 1: ████████  (파일 읽기)
스레드 2: ██████    (네트워크 요청)      ← 동시에 진행
스레드 3: ████      (계산 작업)
```

**싱글 스레드** (JavaScript):

```
메인 스레드: ████ → ██████ → ████      ← 하나씩 순서대로
```

### 멀티 스레드의 장점과 대가

동시에 여러 작업을 처리할 수 있다는 건 강력하지만, **공유 메모리** 때문에 심각한 문제가 생깁니다.

```java
// Java 멀티 스레드 예시 (JS가 아닙니다)
int balance = 10000;

// 스레드 A: 3000원 출금
balance = balance - 3000;  // balance를 읽음 → 10000

// 스레드 B: 동시에 5000원 출금  
balance = balance - 5000;  // balance를 읽음 → 10000 (아직 A가 안 끝남!)

// 결과: 2000이어야 하는데, 5000 또는 7000이 될 수 있음!
```

이것을 **경쟁 조건(Race Condition)**이라 부릅니다. 두 스레드가 같은 데이터를 동시에 건드려서 예측 불가능한 결과가 나오는 것이죠. 이를 방지하려면 **락(Lock)**, **뮤텍스(Mutex)** 같은 동기화 메커니즘이 필요한데, 이게 코드를 매우 복잡하게 만듭니다.

### JavaScript가 싱글 스레드를 선택한 이유

JavaScript는 **브라우저에서 DOM을 조작**하기 위해 태어난 언어입니다.

만약 JS가 멀티 스레드였다면:

```
스레드 A: document.getElementById('btn').remove()   // 버튼 삭제
스레드 B: document.getElementById('btn').click()     // 버튼 클릭 ← 동시에?!
```

같은 DOM 요소를 두 스레드가 동시에 조작하면 어떤 결과가 나올지 예측할 수 없습니다. 이런 복잡성을 원천적으로 제거하기 위해, JS는 **메인 스레드 하나로 모든 코드를 실행**하는 방식을 택했습니다.

### 그런데 "정말" 싱글 스레드일까?

여기서 중요한 구분이 필요합니다:

> **JavaScript "엔진"은 싱글 스레드이지만, JavaScript "런타임 환경"은 아닙니다.**

[[013. 자바스크립트 엔진 vs 자바스크립트 런타임 환경]]

```
┌─────────────────────────────────────────────────┐
│              브라우저 런타임 환경                    │
│                                                   │
│  ┌──────────────┐    ┌────────────────────────┐  │
│  │  JS 엔진(V8)  │    │  Web APIs (브라우저 제공) │  │
│  │              │    │                        │  │
│  │  Call Stack  │    │  - Timer (별도 스레드)    │  │
│  │  (싱글 스레드) │    │  - Network (별도 스레드)  │  │
│  │              │    │  - DOM Events          │  │
│  │  Heap        │    │  - File I/O            │  │
│  └──────────────┘    └────────────────────────┘  │
│                                                   │
│  ┌─────────────┐    ┌─────────────────────────┐  │
│  │ Task Queue   │    │ Microtask Queue         │  │
│  └─────────────┘    └─────────────────────────┘  │
│                                                   │
│          ↕ Event Loop (감시자)                     │
└─────────────────────────────────────────────────┘
```

[[014. Task Queue (Macrotask Queue)]]
[[015. Microtask Queue]]

`setTimeout`을 호출하면, **타이머 카운트다운은 브라우저의 별도 스레드**에서 처리됩니다. 네트워크 요청(`fetch`)도 마찬가지입니다. 하지만 그 결과를 받아서 **콜백을 실행하는 것은 반드시 메인 스레드(Call Stack)**에서 이루어집니다.

이것이 바로 여러분이 공부하고 있는 **이벤트 루프**가 존재하는 이유입니다. 다른 스레드에서 완료된 작업의 콜백을 큐에 넣고, 메인 스레드의 Call Stack이 비었을 때 하나씩 꺼내 실행하는 구조인 것이죠.

## 구체적인 예시

여러분의 키워드 학습 맥락에서, 아래 코드를 직접 실행해보세요:

```js
console.log('1 - 메인 스레드 시작');

setTimeout(() => {
  console.log('2 - 타이머 완료 (어떤 스레드가 시간을 쟀을까?)');
}, 0);

fetch('https://jsonplaceholder.typicode.com/todos/1')
  .then(() => {
    console.log('3 - 네트워크 완료 (어떤 스레드가 요청을 보냈을까?)');
  });

Promise.resolve().then(() => {
  console.log('4 - Promise.resolve (이건 별도 스레드가 필요할까?)');
});

console.log('5 - 메인 스레드 끝');
```

**생각해볼 것**: 1~5가 어떤 순서로 출력될까요? 그리고 각각은 "어디에서" 처리되고 "어디에서" 실행되는 걸까요? "처리"와 "실행"을 구분해서 생각해보면, 싱글 스레드와 이벤트 루프의 관계가 명확해질 것입니다.

내 예상
1,5,2,4,3
이유 
1. 별도의 스레드가 필요한 2,3,4는 나중에 처리되고 순서대로 작성된 1,5가 메인스레드에 먼저 들어가고 출력.
2. 3번은 api를 호출하고 응답을 받아오는 과정이 존재하므로 마지막.
3. 2,4번은 모두 대기시간이 0이므로 같은 순서.

1,5,4,2,3
