## 쉬운 설명부터

**자동차**로 비유해보겠습니다.

- **자바스크립트 엔진** = **엔진(모터)**: 연료(코드)를 받아서 실제로 동력(실행)을 만들어내는 핵심 부품. 엔진만 있으면 돌아가긴 하지만, 그것만으로 도로를 달릴 수는 없습니다.
- **자바스크립트 런타임 환경** = **자동차 전체**: 엔진 + 바퀴 + 핸들 + 계기판 + 도로 인프라. 실제로 "달리기" 위해 필요한 모든 것을 포함한 환경입니다.

엔진은 자동차의 일부이지, 자동차 그 자체가 아닙니다. 마찬가지로, **JS 엔진은 런타임 환경의 일부**입니다.

---
자바스크립트 엔진이 다양한 브라우저에서 돌아가게 하기 위해 V8, SpiderMonkey, JavaScriptCore 로 이루어져 있다는거야?


## 자바스크립트 엔진

### 엔진이 하는 일

자바스크립트 엔진의 역할은 명확합니다: **자바스크립트 코드를 파싱하고, 컴파일하고, 실행하는 것.**

오직 이것뿐입니다. `setTimeout`도 모르고, `fetch`도 모르고, `DOM`도 모릅니다.

### 대표적인 엔진들

|엔진|개발사|사용처|
|---|---|---|
|**V8**|Google|Chrome, Node.js, Deno|
|**SpiderMonkey**|Mozilla|Firefox|
|**JavaScriptCore (Nitro)**|Apple|Safari, Bun|

### 엔진의 내부 구조

V8 엔진을 기준으로 보겠습니다:

```
┌─────────────────────────────────────────────────┐
│                    V8 엔진                       │
│                                                  │
│   소스코드                                        │
│     │                                            │
│     ▼                                            │
│  ┌──────────┐     자바스크립트 코드를 읽고          │
│  │  Parser   │     AST(추상 구문 트리)로 변환       │
│  └────┬─────┘                                    │
│       ▼                                          │
│  ┌──────────┐     AST를 바이트코드로 변환하여       │
│  │ Ignition  │     빠르게 실행 (인터프리터)          │
│  │(인터프리터)│                                    │
│  └────┬─────┘                                    │
│       │ "이 함수 자주 호출되네?" (Hot Code 감지)     │
│       ▼                                          │
│  ┌──────────┐     자주 실행되는 코드를              │
│  │TurboFan   │     최적화된 기계어로 컴파일          │
│  │(JIT 컴파일)│     (JIT: Just-In-Time)           │
│  └──────────┘                                    │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │          Memory Management                 │  │
│  │                                            │  │
│  │  ┌─────────────┐    ┌──────────────────┐  │  │
│  │  │  Call Stack   │    │      Heap        │  │  │
│  │  │  (실행 컨텍스트) │    │  (객체, 배열 등)  │  │  │
│  │  └─────────────┘    └──────────────────┘  │  │
│  │                          ↑                 │  │
│  │                   Garbage Collector        │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

엔진이 관리하는 것:

- **Call Stack** — 함수 실행 컨텍스트를 쌓고 빼는 곳
- **Heap** — 객체 등 참조 타입 데이터가 저장되는 곳
- **가비지 컬렉터** — Heap에서 더 이상 참조되지 않는 객체를 정리

### 엔진이 **모르는** 것

```js
// 엔진이 아는 것 ✅
const a = 1 + 2;
function greet() { return 'hello'; }
const obj = { x: 10 };

// 엔진이 모르는 것 ❌ — 이것들은 누군가가 제공해줘야 한다
setTimeout(() => {}, 1000);     // 타이머? 그게 뭔데?
fetch('https://api.com/data');  // 네트워크? 나는 코드만 실행하는데?
document.getElementById('app'); // DOM? 나는 브라우저가 뭔지도 모르는데?
console.log('hello');           // console? 이것도 내 거 아닌데?
```

그렇다면 `setTimeout`, `fetch`, `document`, `console`은 어디서 오는 걸까요?

### 핵심 차이: 같은 엔진, 다른 런타임

||브라우저|Node.js|
|---|---|---|
|**엔진**|V8 (Chrome 기준)|V8 (동일!)|
|**DOM API**|O (`document`, `window`)|X (없음)|
|**파일 시스템**|X (보안상 제한)|O (`fs` 모듈)|
|**이벤트 루프**|브라우저 자체 구현|libuv 라이브러리|
|**Timer**|Web API로 제공|libuv로 제공|
|**렌더링 엔진**|O (Blink 등)|X (화면이 없음)|
|**console**|브라우저 DevTools로 출력|터미널 stdout으로 출력|

같은 `console.log('hello')`라도:

- 브라우저에서는 **브라우저 런타임**이 DevTools 콘솔에 출력
- Node.js에서는 **Node.js 런타임**이 터미널에 출력

**엔진(V8)은 `console`이 뭔지 모릅니다.** 각 런타임 환경이 `console` 객체를 만들어서 엔진의 전역 스코프에 주입해주는 것입니다.

## 엔진과 런타임의 협업 과정

`setTimeout`이 호출될 때 무슨 일이 벌어지는지 흐름을 따라가보겠습니다:

```js
console.log('A');

setTimeout(() => {
  console.log('B');
}, 1000);

console.log('C');
```

```
시간 →

[JS 엔진 - Call Stack]              [런타임 환경]

1. console.log('A') 실행            
   → 'A' 출력                       
   → Stack에서 제거                  

2. setTimeout(cb, 1000) 실행         
   → 엔진: "setTimeout? 나는 모르는 함수인데...
      아, 런타임이 전역에 등록해준 거구나"
   → 런타임에 타이머 등록 요청 ──────→  3. 런타임: 타이머 시작 (별도 스레드)
   → setTimeout이 Stack에서 제거          ⏱️ 카운트다운 1000ms...

3. console.log('C') 실행             
   → 'C' 출력                        
   → Stack에서 제거                   

4. Call Stack 비어있음                   ⏱️ 아직 카운트 중...
   (엔진: 할 일 없음)                

                                     6. 1000ms 경과!
                                        런타임: 콜백을 Task Queue에 넣음
                                        [Task Queue: cb]

                                     7. Event Loop: "Call Stack 비었네?
                                        Queue에서 cb 꺼내서 Stack에 올림"

5. cb() 실행 ←─────────────────────  
   → console.log('B')               
   → 'B' 출력                       
   → Stack에서 제거                  
```

정리하면:

- **1, 4, 8번** (코드 실행) → **JS 엔진**이 담당
- **3번** (타이머 카운트다운) → **런타임 환경**이 담당 (별도 스레드)
- **6, 7번** (큐 관리, 이벤트 루프) → **런타임 환경**이 담당

## 구체적인 예시

이 구분을 체감하기 위해, 아래 코드를 **브라우저 콘솔**과 **Node.js 터미널** 양쪽에서 실행해보세요:

```js
// 양쪽 다 동작하는 코드
console.log(typeof setTimeout);   // ?
console.log(typeof Promise);      // ?
console.log(typeof queueMicrotask); // ?

// 브라우저에서만 동작하는 코드
console.log(typeof window);       // ?
console.log(typeof document);     // ?

// Node.js에서만 동작하는 코드
console.log(typeof global);       // ?
console.log(typeof process);      // ?
console.log(typeof require);      // ?
```

**생각해볼 것**: `setTimeout`은 양쪽 다 있고, `document`는 브라우저에만, `process`는 Node.js에만 있습니다. 세 가지 모두 JS 엔진(V8)이 제공하는 게 아닌데 — 그렇다면 `setTimeout`은 어째서 양쪽 모두에서 쓸 수 있는 걸까요? "같은 이름이지만 제공자가 다르다"는 것이 무슨 의미인지 생각해보면, 엔진과 런타임의 경계가 더 선명해질 것입니다.

### 공부 전 
자바스크립트 엔진 : 브라우저에서 자바스크립트 코드를 실행시키기 위한 툴
자바스크립트 런타임 환경 : 자바스크립트 엔진을 포함하여 fetch, Promise, console, setTimeout 등을 가지고있는 다른 Web API도 모두 존재하는 넓은 환경. 자바스크립트 엔진은 싱글 스레드이나, 런타임 환경은 자바스크립트 엔진 말고도 다른 스레드가 존재하므로 스레드가 여러개 존재한다.
예시 답변 : 어? 자바스크립트 엔진에서 setTimeout의 타입을 어떻게 알고있는건지 궁금하네? 그냥 존재만 알고있는건가...?


## 현재 이해도 진단

|내용|진단|
|---|---|
|엔진 = 코드를 실행시키기 위한 툴|맞음. 다만 "브라우저에서"라는 한정이 부정확 — Node.js에서도 동일한 V8 엔진이 동작|
|런타임 = 엔진 + Web API가 존재하는 넓은 환경|정확함|
|엔진은 싱글 스레드, 런타임은 멀티 스레드|핵심을 정확히 잡고 있음|
|Promise를 Web API라고 분류|부정확 — 아래에서 설명|
|예시 질문: 엔진이 setTimeout 타입을 어떻게 아는지|매우 좋은 질문. 이걸 파고들면 됨|

## 집중해서 공부할 포인트 2가지

### 1. Promise는 Web API가 아니다

```
Web API (런타임이 제공):        JS 엔진이 아는 것 (ECMAScript 명세):
  - setTimeout                  - Promise  ← 여기!
  - fetch                       - let, const, function
  - DOM (document)              - async/await
  - console                     - Array, Object 등
```

`Promise`는 **ECMAScript 표준에 정의된 기능**이라 엔진 자체가 알고 있습니다. `fetch`는 Web API이지만, `fetch`가 **반환하는 Promise**는 엔진이 처리합니다.

이 구분을 자료의 "엔진이 아는 것 / 모르는 것" 섹션에서 다시 확인해보세요.

### 2. 예시 질문에 대한 학습 방향

"엔진이 setTimeout의 타입을 어떻게 아는지"라는 질문에 대한 답은 자료의 **"엔진과 런타임의 협업 과정"** 섹션에 있습니다. 핵심만 힌트를 드리면:

```
프로그램 시작 시:

런타임 환경이 하는 일:
  전역 스코프에 setTimeout, console, fetch 등을 "등록"해줌
  ↓
JS 엔진 입장:
  "전역 스코프에 setTimeout이라는 함수가 있네? 호출해볼게"
  → 호출하면 런타임 환경이 실제 처리를 담당
```

엔진이 setTimeout을 **"알고 있는"** 것이 아니라, 런타임이 **"주입해준"** 것입니다.

## 추천 학습 순서

1. 자료에서 **"엔진이 아는 것 / 모르는 것"** 코드 예시를 읽으면서, Promise가 어느 쪽에 있는지 확인
2. **"엔진과 런타임의 협업 과정"** 의 setTimeout 흐름도를 따라가면서, "주입"이 어떻게 이루어지는지 이해
3. 그 후 스스로 `fetch('/api').then(cb)`가 실행될 때 **어디까지가 엔진의 일이고, 어디부터가 런타임의 일인지** 경계선을 그어보세요
