## 쉬운 설명부터

**서류 작업 트레이**를 상상해보세요.

```
사무실에서 일하는 중:

1. "보고서 A" 작업 중 (책상 위에 A 관련 서류)
2. 급한 전화 → "보고서 B" 처리해야 함
   → A 서류를 트레이에 넣고, B 서류를 꺼냄
3. B 완료 → B 서류 치우고, 트레이에서 A 서류 꺼냄
4. A 이어서 작업
```

- **서류 한 묶음** = 실행 컨텍스트 (그 작업을 하기 위한 모든 정보)
- **트레이(쌓기)** = Call Stack
- **책상 위** = 현재 실행 중인 컨텍스트

실행 컨텍스트는 **"이 코드를 실행하기 위해 필요한 모든 정보를 묶어놓은 환경"**입니다.

---

## 자세한 설명

### 실행 컨텍스트란?

> 코드가 실행되기 위해 필요한 **변수, 함수 선언, 스코프 체인, this** 등의 정보를 담고 있는 객체(환경)

자바스크립트 엔진은 코드를 실행할 때, 무턱대고 한 줄씩 읽는 것이 아닙니다. 먼저 **"이 코드를 실행하려면 뭐가 필요하지?"**를 정리한 환경을 만들고, 그 안에서 실행합니다.

### 실행 컨텍스트가 생성되는 3가지 경우

```js
// 1. 전역 코드가 실행될 때 → 전역 실행 컨텍스트
const name = "Lee";
console.log(name);

// 2. 함수가 호출될 때 → 함수 실행 컨텍스트
function greet() {
  const message = "hello";
  console.log(message);
}
greet(); // ← 이 순간 새로운 실행 컨텍스트 생성

// 3. eval()이 실행될 때 → eval 실행 컨텍스트 (사용 비권장)
```

**함수를 선언할 때가 아니라, 호출할 때** 생성된다는 점이 중요합니다.

### 실행 컨텍스트 스택 (Call Stack)

이전에 배운 **Call Stack**이 바로 실행 컨텍스트가 쌓이는 곳입니다.

```js
function first() {
  console.log("first 시작");
  second();
  console.log("first 끝");
}

function second() {
  console.log("second 시작");
  third();
  console.log("second 끝");
}

function third() {
  console.log("third");
}

first();
```

```
       Call Stack 변화 과정

  ①              ②              ③              ④
│          │  │          │  │          │  │          │
│          │  │          │  │  third   │  │          │
│          │  │  second  │  │  second  │  │  second  │
│  first   │  │  first   │  │  first   │  │  first   │
│  전역     │  │  전역     │  │  전역     │  │  전역     │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
first() 호출   second() 호출  third() 호출   third() 끝
                                          → pop

  ⑤              ⑥              ⑦
│          │  │          │  │          │
│          │  │          │  │          │
│          │  │          │  │          │
│  first   │  │          │  │          │
│  전역     │  │  전역     │  │          │
└──────────┘  └──────────┘  └──────────┘
second() 끝    first() 끝     프로그램 종료
→ pop          → pop          전역도 pop
```

출력 순서:

```
"first 시작" → "second 시작" → "third" → "second 끝" → "first 끝"
```

**스택의 가장 위(top)에 있는 컨텍스트가 현재 실행 중인 코드**입니다. 항상 하나만 실행 중이죠 — 이것이 **싱글 스레드**의 의미입니다.

### 실행 컨텍스트의 내부 구조

하나의 실행 컨텍스트 안에는 무엇이 들어있을까요?

```
┌─────────────────────────────────────────────────┐
│            Execution Context (실행 컨텍스트)       │
│                                                   │
│  ┌─────────────────────────────────────────────┐ │
│  │  Lexical Environment (렉시컬 환경)            │ │
│  │                                              │ │
│  │  ┌────────────────────────────────────────┐  │ │
│  │  │ Environment Record (환경 레코드)         │  │ │
│  │  │                                        │  │ │
│  │  │  - 이 스코프의 변수/함수 선언 저장        │  │ │
│  │  │  - let, const, function 등              │  │ │
│  │  └────────────────────────────────────────┘  │ │
│  │                                              │ │
│  │  ┌────────────────────────────────────────┐  │ │
│  │  │ Outer Lexical Environment Reference     │  │ │
│  │  │ (외부 렉시컬 환경 참조)                    │  │ │
│  │  │                                        │  │ │
│  │  │  - 상위 스코프를 가리키는 포인터           │  │ │
│  │  │  - 스코프 체인의 핵심!                    │  │ │
│  │  └────────────────────────────────────────┘  │ │
│  └─────────────────────────────────────────────┘ │
│                                                   │
│  ┌─────────────────────────────────────────────┐ │
│  │  This Binding (this 바인딩)                   │ │
│  │  - 이 컨텍스트에서 this가 가리키는 대상        │ │
│  └─────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

코드로 살펴보겠습니다:

```js
const globalVar = "전역";

function outer() {
  const outerVar = "외부";

  function inner() {
    const innerVar = "내부";
    console.log(innerVar); // '내부'   ← 자기 환경 레코드에서 찾음
    console.log(outerVar); // '외부'   ← 외부 환경 참조를 따라 outer에서 찾음
    console.log(globalVar); // '전역'   ← 더 따라가서 전역에서 찾음
  }

  inner();
}

outer();
```

```
inner의 실행 컨텍스트:
  환경 레코드: { innerVar: '내부' }
  외부 환경 참조: → outer의 렉시컬 환경
  this: window

outer의 실행 컨텍스트:
  환경 레코드: { outerVar: '외부', inner: function }
  외부 환경 참조: → 전역 렉시컬 환경
  this: window

전역 실행 컨텍스트:
  환경 레코드: { globalVar: '전역', outer: function }
  외부 환경 참조: → null (더 이상 바깥이 없음)
  this: window
```

`console.log(outerVar)`를 inner 안에서 실행하면:

```
inner 환경 레코드에서 검색 → 없음
  → 외부 환경 참조를 따라 outer로 이동
    → outer 환경 레코드에서 검색 → 찾음! '외부'
```

이것이 **스코프 체인**입니다. 외부 환경 참조를 따라 연쇄적으로 상위 스코프를 탐색하는 구조.

### 실행 컨텍스트의 2단계 처리

코드를 실행할 때 **평가(Creation Phase)**와 **실행(Execution Phase)**, 두 단계로 나뉩니다.

```js
console.log(x); // undefined (에러 아님!)
console.log(y); // ReferenceError!
console.log(greet); // function greet() { ... } (함수 전체!)

var x = 10;
let y = 20;
function greet() {
  return "hi";
}
```

**1단계 — 평가 (Creation Phase)**: 코드를 실행하기 전에 환경 레코드를 먼저 구성

```
환경 레코드 구성:
  x: undefined              ← var → 선언 + undefined로 초기화
  y: <uninitialized>        ← let → 선언만, 초기화 안 됨 (TDZ)
  greet: function() {...}   ← 함수 선언문 → 선언 + 함수 전체가 초기화
```

**2단계 — 실행 (Execution Phase)**: 위에서 아래로 코드를 한 줄씩 실행

```
console.log(x)     → 환경 레코드에서 x 찾음 → undefined (초기화된 상태)
console.log(y)     → 환경 레코드에서 y 찾음 → <uninitialized> → ReferenceError!
console.log(greet) → 환경 레코드에서 greet 찾음 → function 전체

var x = 10         → x에 10 할당
let y = 20         → y에 20 할당 (이제야 초기화)
```

이것이 **호이스팅(hoisting)**의 실체입니다. 코드가 "끌어올려지는" 것이 아니라, **평가 단계에서 환경 레코드에 먼저 등록**되기 때문에 선언 전에 접근할 수 있는(또는 없는) 것입니다.

```
var:      선언 + undefined 초기화  → 선언 전 접근 시 undefined
let/const: 선언만                  → 선언 전 접근 시 ReferenceError (TDZ)
함수 선언문: 선언 + 함수 전체 초기화  → 선언 전 접근 시 함수 사용 가능
```

## 이벤트 루프 관점에서 실행 컨텍스트

이벤트 루프가 Task Queue에서 콜백을 꺼내서 실행할 때, **새로운 실행 컨텍스트가 생성**됩니다.

```js
console.log("A");

setTimeout(() => {
  const x = 10;
  console.log("B:", x);
}, 0);

console.log("C");
```

```
시점 1: 전역 코드 실행
┌──────────┐
│  전역 EC   │  ← console.log('A'), setTimeout 등록, console.log('C')
└──────────┘

시점 2: 전역 코드 끝 → 전역 EC는 남아있지만 실행 완료

시점 3: Event Loop가 콜백을 Stack에 올림
┌──────────┐
│ 콜백 EC    │  ← const x = 10, console.log('B:', x)
│            │    환경 레코드: { x: 10 }
│            │    외부 환경 참조: → 전역 렉시컬 환경
└──────────┘

시점 4: 콜백 실행 완료 → 콜백 EC pop → Stack 비어짐
```

**각 콜백이 실행될 때마다 독립적인 실행 컨텍스트가 생성**됩니다. 이 컨텍스트 안에 그 콜백의 변수, 스코프 체인, this가 모두 담겨있는 것입니다.

## 구체적인 예시

아래 코드의 실행 과정을 Call Stack과 환경 레코드를 그려가며 추적해보세요:

```js
var a = 1;

function outer() {
  var b = 2;
  console.log(a); // ?

  function inner() {
    var c = 3;
    console.log(a); // ?
    console.log(b); // ?
  }

  inner();
  console.log(c); // ?
}

outer();
console.log(b); // ?
```

**생각해볼 포인트**:

1. 각 `console.log`에서 값을 찾을 때, **어떤 환경 레코드를 먼저 보고, 어디로 올라가는지** 경로를 추적해보세요
2. `console.log(c)`와 `console.log(b)`에서 에러가 나는 이유를 — "변수가 없어서"가 아니라 **"스코프 체인을 따라 올라가도 해당 환경 레코드에 등록되지 않았기 때문"**이라고 설명할 수 있어야 합니다
3. `inner()`가 호출되는 시점의 Call Stack에 쌓인 실행 컨텍스트를 아래부터 순서대로 나열해보세요

### 공부후

실행컨텍스트 : 콜스택에 올라가는 객체(환경)로 변수나 함수등 해당 코드의 실행환경이 저장되어있음.

질문 :

- 콜스택 자체가 자바스크립트 엔진의 싱글 스레드인가? 혹은 콜스택과 힙메모리까지 합쳐서 싱글 스레드로 보는건가?
- 자바스크립트런타임 환경이 가장 큰 범주고 그 안에 다양한 환경이 존재하고 있는건가?
- 왜 함수는 구체적 내용까지 함께 올라가는건가? 원래 다 올리고싶은데 값이 할당되지 않은 변수라 이름만 올리는건가?
