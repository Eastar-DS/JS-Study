## 쉬운 설명부터

**전화번호를 남기는 것**과 같습니다.

```
병원에 전화: "예약 자리 나면 이 번호로 연락주세요"
               ─────────────────────────────
               이 번호 = 콜백 함수
               연락주세요 = 나중에 호출해주세요
```

- 내가 직접 전화를 거는 게 아니라, **"이 번호로 다시(back) 전화해줘(call)"**라고 맡기는 것
- 언제 전화가 올지는 **병원(호출자)**이 결정

이것이 콜백(callback)의 어원입니다. **Call(호출) + Back(되돌려)** = 나중에 되돌려 호출해줘.

## 자세한 설명

### 콜백 함수의 정의

> **다른 함수에 인수로 전달되어, 그 함수에 의해 나중에 호출되는 함수**

```js
// 가장 단순한 형태
function greeting(name) {
  console.log(`안녕, ${name}!`);
}

function processUser(callback) {
  const userName = "김개발";
  callback(userName); // 전달받은 함수를 호출
}

processUser(greeting); // greeting 함수를 "인수로 전달"
// 출력: "안녕, 김개발!"
```

여기서 핵심은 **제어권의 이전**입니다.

```
일반 함수 호출:        내가 직접 호출
  greeting('김개발')    → 내가 호출 시점, 인수를 결정

콜백으로 전달:          남에게 맡김
  processUser(greeting) → processUser가 호출 시점, 인수를 결정
```

### 콜백 함수가 가능한 이유 — 일급 객체

자바스크립트에서 함수는 **일급 객체(First-class citizen)**입니다. 즉, 함수를 **값처럼** 다룰 수 있습니다.

```js
// 변수에 담을 수 있다
const fn = function () {
  return 1;
};

// 다른 함수의 인수로 전달할 수 있다
setTimeout(fn, 1000);

// 다른 함수의 반환값이 될 수 있다
function createGreeter() {
  return function (name) {
    console.log(`Hi, ${name}`);
  };
}
```

함수를 값처럼 주고받을 수 있기 때문에, "함수를 인수로 넘기고 나중에 호출" 하는 콜백 패턴이 가능한 것입니다.

### 제어권의 3가지 — 코어 자바스크립트 관점

콜백 함수를 넘기면, **3가지 제어권**을 상대에게 위임합니다.

#### 1. 호출 시점

```js
// setInterval이 300ms마다 호출 시점을 결정
let count = 0;
const timerId = setInterval(() => {
  console.log(++count);
  if (count >= 3) clearInterval(timerId);
}, 300);

// 내가 아닌 setInterval이 "언제" 콜백을 호출할지 결정
```

#### 2. 인수

```js
// map이 콜백에 전달하는 인수를 결정
const result = [10, 20, 30].map((currentValue, index, array) => {
  //                             ↑ map이 정한 순서: 값, 인덱스, 배열
  return currentValue * 2;
});
```

**인수의 순서와 의미는 콜백을 호출하는 쪽이 정합니다.** 내가 매개변수 이름을 바꿔봤자 소용없습니다:

```js
// ❌ 매개변수 이름을 index라고 지어도, 첫 번째 인수는 여전히 "값"
[10, 20, 30].map((index) => {
  console.log(index); // 10, 20, 30 (인덱스가 아니라 값!)
});
```

#### 3. this

```js
// addEventListener가 this를 해당 DOM 요소로 바인딩
document.getElementById("btn").addEventListener("click", function () {
  console.log(this); // <button id="btn">... (DOM 요소)
});

// 반면 setTimeout은 this를 window(전역)로
setTimeout(function () {
  console.log(this); // window (브라우저) / global (Node.js)
}, 100);
```

`this`가 무엇이 될지도 **콜백을 호출하는 쪽**이 결정합니다. 같은 함수라도 누가 호출하느냐에 따라 `this`가 달라집니다.

## 콜백의 두 가지 용도

### 1. 동기 콜백 — 즉시 실행

```js
// forEach: 콜백을 지금 당장, 반복적으로 호출
["사과", "바나나", "체리"].forEach((fruit) => {
  console.log(fruit);
});
// "사과" → "바나나" → "체리" (즉시, 동기적으로)

// sort: 콜백을 비교 기준으로 즉시 사용
[3, 1, 2].sort((a, b) => a - b);

// filter, map, reduce 등 모두 동기 콜백
```

동기 콜백은 **"어떻게 처리할지"를 함수로 전달**하는 패턴입니다. 호출 즉시 실행되고 끝납니다.

### 2. 비동기 콜백 — 나중에 실행

```js
// setTimeout: 일정 시간 후에 호출
setTimeout(() => {
  console.log("3초 뒤");
}, 3000);

// addEventListener: 이벤트 발생 시 호출
button.addEventListener("click", () => {
  console.log("클릭됨");
});

// fetch의 .then: 응답 도착 시 호출
fetch("/api/data").then((response) => {
  console.log("응답 도착");
});
```

비동기 콜백은 **"나중에 뭘 할지"를 함수로 전달**하는 패턴입니다. 이것이 이벤트 루프와 직결됩니다.

## 이벤트 루프 관점에서 콜백

이벤트 루프를 공부하는 맥락에서 핵심 질문: **"콜백이 등록되고 실행되기까지 어디에 있는가?"**

```js
console.log("A");

button.addEventListener("click", function handler() {
  console.log("B");
});

setTimeout(function timer() {
  console.log("C");
}, 1000);

fetch("/api").then(function network() {
  console.log("D");
});

console.log("E");
```

```
등록 시점                 대기 장소                  실행 트리거

handler  ──→  Web API (이벤트 리스너 목록)  ──→  클릭 발생 시 Task Queue에
timer    ──→  Web API (타이머 스레드)       ──→  1000ms 후 Task Queue에
network  ──→  Web API (네트워크 스레드)     ──→  응답 도착 시 Microtask Queue에

                                                     │
                                                     ▼
                                            Event Loop가 Call Stack이
                                            비었을 때 꺼내서 실행
```

콜백 함수 자체는 **Heap에 저장**된 함수 객체입니다 (이전에 메모리 공간에서 배운 내용!). Web API가 이 함수 객체의 **참조(주소)**를 들고 있다가, 조건이 충족되면 Queue에 넣는 것입니다.

### 콜백 지옥 — 비동기 콜백의 한계

비동기 작업이 순차적으로 이어져야 할 때 문제가 드러납니다:

```js
// 1단계가 끝나야 2단계, 2단계가 끝나야 3단계...
loginUser(
  id,
  password,
  (user) => {
    getProfile(
      user.id,
      (profile) => {
        getSettings(
          profile.settingId,
          (settings) => {
            updateTheme(
              settings.theme,
              (result) => {
                renderDashboard(
                  result,
                  (html) => {
                    console.log("드디어 완료!");
                    // 여기서 에러 나면? 바깥으로 전파가 안 됨...
                  },
                  handleError,
                );
              },
              handleError,
            );
          },
          handleError,
        );
      },
      handleError,
    );
  },
  handleError,
);
```

문제점:

1. **가독성**: 들여쓰기가 계속 깊어짐
2. **에러 처리**: 각 단계마다 개별 처리해야 함
3. **흐름 제어**: 여러 비동기 작업을 병렬로 실행하거나 경쟁시키기 어려움

이 한계를 해결하기 위해 등장한 것이 이전에 배운 **Promise**입니다:

```js
loginUser(id, password)
  .then((user) => getProfile(user.id))
  .then((profile) => getSettings(profile.settingId))
  .then((settings) => updateTheme(settings.theme))
  .then((result) => renderDashboard(result))
  .then((html) => console.log("완료!"))
  .catch((error) => handleError(error)); // 에러 한 곳에서 처리
```

## 구체적인 예시

아래 코드를 보고 생각해보세요:

```js
function processData(data, onSuccess, onError) {
  if (!data) {
    onError("데이터 없음");
    return;
  }
  const result = data.map((item) => item * 2);
  onSuccess(result);
}

console.log("1");

processData(
  [1, 2, 3],
  (result) => console.log("2:", result),
  (error) => console.log("3:", error),
);

console.log("4");

setTimeout(() => {
  processData(
    null,
    (result) => console.log("5:", result),
    (error) => console.log("6:", error),
  );
}, 0);

Promise.resolve().then(() => console.log("7"));

console.log("8");
```

**생각해볼 포인트**:

1. `processData`에 전달된 `onSuccess`, `onError`는 콜백 — 그런데 **동기 콜백**일까 **비동기 콜백**일까?
2. 동기 콜백이라면 `'2'`는 `'4'`보다 먼저일까 나중일까?
3. `setTimeout` 안의 `processData` 호출은 — `processData` 자체는 동기인데, `setTimeout`으로 감싸면 어떻게 될까?
4. `'6'`, `'7'` 중 어느 것이 먼저 출력될까? (Microtask vs Macrotask)
