## 쉬운 설명부터

**은행 창구 직원**을 상상해보세요.

```
창구 직원 1명 (= 싱글 스레드)

하는 일:
1. 지금 처리 중인 고객 업무를 끝낸다 (Call Stack)
2. 끝나면 "대기 번호판"을 확인한다
3. VIP 대기열을 먼저 전부 처리한다 (Microtask Queue)
4. VIP 대기열이 비면, 일반 대기열에서 1명 처리한다 (Task Queue)
5. 다시 2번으로 돌아간다

이것을 영원히 반복한다 → "루프(Loop)"
```

이벤트 루프는 **"Call Stack이 비었니? 그러면 Queue에서 다음 작업을 꺼내올게"를 끊임없이 반복하는 감시자**입니다.

---

## 자세한 설명

### 이벤트 루프의 존재 이유

1단계에서 배운 내용을 복기해봅시다:

- JS 엔진은 **싱글 스레드** → Call Stack 하나로 코드를 실행
- 그런데 `setTimeout`, `fetch` 같은 비동기 작업은 **런타임 환경(Web API)이 별도 스레드**에서 처리
- 별도 스레드에서 완료된 작업의 콜백을 **다시 Call Stack에 올려줄 누군가**가 필요

그 "누군가"가 바로 **이벤트 루프**입니다.

```
┌──────────────────────────────────────────────────────────┐
│                    런타임 환경                              │
│                                                          │
│  ┌──────────┐                    ┌─────────────────┐     │
│  │ Call Stack │ ←── ② 꺼내서     │    Web APIs      │     │
│  │           │     Stack에 올림   │  (별도 스레드)     │     │
│  └──────────┘                    │                 │     │
│       │                          │ setTimeout 타이머│     │
│       │ ① Stack                  │ fetch 네트워크   │     │
│       │   비었나?                 │ DOM 이벤트 감지  │     │
│       │                          └────────┬────────┘     │
│       │                                   │              │
│  ┌────▼──────────────────┐                │ ③ 완료되면   │
│  │     Event Loop         │                │   Queue에    │
│  │     (무한 반복 감시자)    │                │   넣음       │
│  │                        │                │              │
│  │  "Stack 비었으면        │   ┌────────────▼──────────┐  │
│  │   Queue에서 꺼내!"      │   │                       │  │
│  │                        │   │  Microtask Queue (VIP) │  │
│  │                        │   │  Task Queue (일반)      │  │
│  └────────────────────────┘   └───────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

### 이벤트 루프의 알고리즘

의사 코드(pseudocode)로 표현하면:

```
while (true) {   // 무한 루프 — 이래서 "이벤트 루프"

  // 1. Call Stack이 비어있는지 확인
  if (callStack.isEmpty()) {

    // 2. Microtask Queue를 전부 소진
    while (microtaskQueue.hasTask()) {
      const task = microtaskQueue.dequeue();
      callStack.push(task);
      execute(task);
    }

    // 3. 렌더링이 필요하면 렌더링 (브라우저 한정)
    if (needsRender()) {
      runRequestAnimationFrameCallbacks();
      render();  // Style → Layout → Paint → Composite
    }

    // 4. Task Queue에서 1개만 꺼내서 실행
    if (taskQueue.hasTask()) {
      const task = taskQueue.dequeue();
      callStack.push(task);
      execute(task);
    }
  }
}
```

핵심 규칙 3가지:

```
규칙 1: Call Stack이 비어있어야만 Queue의 작업을 가져온다
규칙 2: Microtask Queue를 "전부" 소진한 후에야 다른 일을 한다
규칙 3: Task Queue에서는 "1개만" 꺼내고, 다시 Microtask 확인으로 돌아간다
```

### Task Queue vs Microtask Queue

```
┌──────────────────────────────────────────────────────┐
│  Microtask Queue (VIP 대기열) — 먼저, 전부 소진        │
│                                                      │
│  • Promise.then / .catch / .finally                  │
│  • queueMicrotask()                                  │
│  • MutationObserver                                  │
│  • async/await (await 이후 코드)                      │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│  Task Queue (일반 대기열) — Microtask 후, 1개씩        │
│  (= Macrotask Queue)                                 │
│                                                      │
│  • setTimeout / setInterval                          │
│  • I/O 이벤트                                         │
│  • UI 이벤트 (click, scroll 등)                       │
│  • MessageChannel                                    │
└──────────────────────────────────────────────────────┘
```

### 한 사이클의 전체 흐름

```
         ┌─────────────────────┐
         │  Task Queue에서      │
    ┌───→│  Macrotask 1개 실행   │
    │    └─────────┬───────────┘
    │              │
    │              ▼
    │    ┌─────────────────────┐
    │    │  Microtask Queue     │
    │    │  전부 소진            │◄──── Microtask 안에서 새 Microtask가
    │    │  (새로 추가된 것 포함)  │       생겨도 여기서 다 처리
    │    └─────────┬───────────┘
    │              │
    │              ▼
    │    ┌─────────────────────┐
    │    │  렌더링 기회           │
    │    │  (필요하면 렌더링)      │  ← rAF 콜백도 여기서 실행
    │    └─────────┬───────────┘
    │              │
    └──────────────┘
          다음 사이클
```

## 단계별 실행 추적

### 기본 예제

```js
console.log("1");

setTimeout(() => {
  console.log("2");
}, 0);

Promise.resolve().then(() => {
  console.log("3");
});

console.log("4");
```

한 줄씩 추적:

```
── 동기 코드 실행 ──

console.log('1')                                          출력: "1"
  → Stack: [전역]

setTimeout(cb, 0)
  → Web API에 타이머 등록 → 0ms 후 cb를 Task Queue에
  → Task Queue: [cb-timeout]

Promise.resolve().then(cb2)
  → 즉시 fulfilled → cb2를 Microtask Queue에
  → Microtask Queue: [cb2-promise]

console.log('4')                                          출력: "4"

── 동기 코드 끝, Call Stack 비어짐 ──

── Event Loop: Microtask Queue 확인 ──
cb2-promise 실행 → console.log('3')                       출력: "3"
  Microtask Queue: [] (비어짐)

── Event Loop: 렌더링 기회 (패스) ──

── Event Loop: Task Queue 확인 ──
cb-timeout 실행 → console.log('2')                        출력: "2"
  Task Queue: [] (비어짐)

최종 출력: "1" → "4" → "3" → "2"
```

### Microtask가 Microtask를 생성하는 경우

```js
setTimeout(() => console.log("A"), 0);

Promise.resolve().then(() => {
  console.log("B");
  Promise.resolve().then(() => {
    console.log("C");
  });
});

Promise.resolve().then(() => {
  console.log("D");
});
```

```
── 동기 코드 끝 ──

Microtask Queue: [cb-B, cb-D]
Task Queue:      [cb-A]

── Microtask 소진 시작 ──

cb-B 실행 → "B" 출력
  → 내부에서 Promise.resolve().then(cb-C) 등록
  Microtask Queue: [cb-D, cb-C]  ← cb-C가 추가됨!

cb-D 실행 → "D" 출력
  Microtask Queue: [cb-C]

cb-C 실행 → "C" 출력              ← 새로 추가된 것도 지금 처리!
  Microtask Queue: []

── Microtask 전부 소진 완료! ──

── Task Queue ──
cb-A 실행 → "A" 출력

출력: "B" → "D" → "C" → "A"
```

**Macrotask(A)는 Microtask가 모두 소진된 후에야 실행됩니다.** 중간에 생겨난 cb-C까지 다 처리한 후에.

### Macrotask → Microtask → Macrotask 교차

```js
setTimeout(() => {
  console.log("T1");
  Promise.resolve().then(() => console.log("M1"));
}, 0);

setTimeout(() => {
  console.log("T2");
  Promise.resolve().then(() => console.log("M2"));
}, 0);
```

```
── 동기 코드 끝 ──

Task Queue: [cb-T1, cb-T2]

── 사이클 1 ──
Task에서 1개: cb-T1 실행 → "T1" 출력
  → 내부에서 Microtask 등록
  Microtask Queue: [cb-M1]

Microtask 소진: cb-M1 실행 → "M1" 출력

(렌더링 기회)

── 사이클 2 ──
Task에서 1개: cb-T2 실행 → "T2" 출력
  → 내부에서 Microtask 등록
  Microtask Queue: [cb-M2]

Microtask 소진: cb-M2 실행 → "M2" 출력

출력: "T1" → "M1" → "T2" → "M2"
```

이것이 **"Macrotask 1개 실행 → Microtask 전부 소진 → 다음 Macrotask"** 사이클의 실체입니다.

## 구체적인 예시

아래 코드의 출력 순서를 예측해보세요:

```js
console.log("1");

setTimeout(() => {
  console.log("2");
  Promise.resolve().then(() => console.log("3"));
}, 0);

new Promise((resolve) => {
  console.log("4");
  resolve();
})
  .then(() => {
    console.log("5");
    setTimeout(() => console.log("6"), 0);
  })
  .then(() => {
    console.log("7");
  });

setTimeout(() => {
  console.log("8");
}, 0);

console.log("9");
```

**생각해볼 포인트**:

1. `new Promise(executor)` 안의 `console.log('4')`는 **동기 실행** — 어디에 위치할까?
2. 동기 코드가 다 끝난 시점에 Microtask Queue와 Task Queue에 각각 무엇이 들어있을까?
3. `.then(() => '5')` 안에서 등록한 `setTimeout(() => '6')`은 **어느 사이클**에서 실행될까?
4. `.then(() => '7')`은 `'5'`의 `.then()`이 반환한 Promise가 fulfilled된 후에 등록됨 — 그렇다면 `'5'`와 `'7'` 사이에 다른 것이 끼어들 수 있을까?
5. Task Queue의 `'2'`와 `'8'` 사이에서 이벤트 루프는 매번 Microtask Queue를 확인한다

종이에 **Call Stack / Microtask Queue / Task Queue** 세 칸을 그리고, 한 줄씩 상태 변화를 추적해보세요.
